# FVG-уведомления 15m

## Правило сигнала

Модуль использует классическую конструкцию из трёх последовательных свечей
`A, B, C` на 15 минутах:

- бычий FVG: `low(C) > high(A)`, зона `[high(A), low(C)]`;
- медвежий FVG: `high(C) < low(A)`, зона `[high(C), low(A)]`.

Подтверждённый сигнал рассчитывается только по закрытым и полным свечам.
Предварительный сигнал проверяется один раз в точке T−3: после первых 12 полных
минут свечи C. Если бот запущен позже этого минутного окна, запоздалое
предварительное уведомление не отправляется. Фильтра по объёму нет.

Цена сигнала — `close(C)`: для предварительного события это последняя цена
агрегированной свечи, для подтверждённого — окончательное закрытие C. Границы
ценового фильтра включительные и вычисляются через `Decimal`.

## Поток данных и восстановление

`BitunixFvgStream` поддерживает одно публичное WebSocket-соединение
`wss://fapi.bitunix.com/public/`, официальный JSON heartbeat и по одному
набору подписок `market_kline_1min`/`market_kline_15min` на каждый активный
инструмент. Список подписок обновляется при изменении пользовательских настроек.
После разрыва используется экспоненциальная задержка переподключения.

REST-запросы к `/api/v1/futures/market/kline` выполняются при старте,
переподключении и раз в пять минут как страховочное восстановление. После
восстановления разрешён только последний подтверждённый FVG не старше одного
15-минутного интервала. Пред-FVG задним числом запрещён.

## Настройки и миграция

Настройки хранятся в `data/fvg_alert_settings.json` по Telegram ID. Отдельно
настраиваются общий выключатель, предварительные и подтверждённые уведомления,
бычье и медвежье направление, набор инструментов и ценовой диапазон для каждого
инструмента. Ценовой фильтр можно применять к `PRE_FVG`, `CONFIRMED_FVG` или к
обоим типам.

Старый JSON автоматически читается как схема v2. Существующие включённые
подтверждённые уведомления сохраняются. Предварительные уведомления сохраняются
только для пользователей, которые ранее включили их явно; для остальных они
выключены. Запись в новом формате происходит при следующем изменении настройки.

## Дедупликация

`data/fvg_event_store.json` разделяет:

- рыночное событие с детерминированным `event_id`;
- доставку пары `(event_id, telegram_chat_id)`.

Поэтому одно событие регистрируется один раз, каждый подходящий пользователь
получает его не более одного раза, а перезапуск процесса не создаёт дублей.
Записи выполняются атомарной заменой файла и защищены общей блокировкой пути.

## Наблюдаемость

Команда `/status` показывает состояние WebSocket, активные инструменты, время
последнего WS-сообщения и REST-восстановления, количество событий, доставок,
переподключений и ошибок. Подробные ошибки пишутся в журнал приложения без
API-ключей и пользовательских настроек.
